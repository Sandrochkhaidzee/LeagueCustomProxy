# v0.3.0 Design

**Status:** design approved 2026-06-02. Implementation plan to be produced separately via the `writing-plans` skill. Author: drafted with Claude during the v0.3.0 brainstorming session.

## Goal

Ship a release that (a) reframes proximity audio as a **competitive-friendly default with opt-in extended hearing** rather than a default-leaky feature, (b) fixes two known UX bugs (#11 panel resize, #1 in-game PTT), and (c) addresses CV tracking reliability problems surfaced by v0.1.33 issue #7 logs through code tweaks + a model retrain.

The framing change is the largest. v0.2.x serves cross-team voice at full vision range (1200u) by default; a modified client running v0.2.x can read every enemy's per-tick distance vector inside that range. v0.3.0 narrows the default to ~auto-attack range (600u) and gates the full-vision range behind a Settings toggle, with the team filter enforced server-side so a modified client cannot bypass it.

## 1. Tiered proximity audio

### Volume rules (server-authoritative)

For each peer P in requester R's room:

| Condition | Result |
|---|---|
| `P.team == R.team` | `volume = 1.0` (always-full team voice) |
| `P.team != R.team` AND distance < R's effective range | `volume = max(0, 1 - (d/1200)²)` |
| `P.team != R.team` AND distance ≥ R's effective range | peer omitted from response entirely |

Effective range is **600u by default**, **1200u when the requester's `hearCrossTeam` toggle is on**. The falloff curve uses the **full 1200u denominator** in both cases — only the cutoff distance changes — so a peer at distance X is heard at the same loudness regardless of toggle state. The transition at the cutoff distance is a hard cut, not a fade.

Rationale for hard-cut: gives consistent loudness for in-range peers across all users in the room, gives an honest "they just entered hearing range" signal, and avoids the rescale-to-range pitfall where two users would hear the same enemy at different loudnesses depending on toggle.

### Wire protocol changes

1. WSS `join` message gains a `team` field:
   ```
   { type: "join", room, name, team: "ORDER" | "CHAOS" }
   ```
   Server stores `team` on `ClientInfo` alongside `name`/`ws`/`position`.

2. WSS `coords` message gains a `hearCrossTeam` field (piggyback, not a separate message type):
   ```
   { type: "coords", x, y, hearCrossTeam: true | false }
   ```
   Server updates `hearCrossTeam` on `ClientInfo` on every receive. Toggle changes pick up within ~100ms.

3. `POST /compute-volumes` request body unchanged in shape (`{myPosition, roomId, name}`). Server reads `team` and `hearCrossTeam` from room state, not the request body. Keeps the HTTP path stateless; prevents a client from lying about its team in a single request.

4. `POST /compute-volumes` response unchanged in shape (`{peerVolumes}`). Cross-team peers absent when toggle is off OR out of range; allies always present at 1.0.

### Back-compat

- `join` message without `team` → server falls back to legacy v0.2 behavior (team-blind, 1200u for everyone, both teams audible at full range). Preserves v0.2.x client experience exactly.
- `coords` message without `hearCrossTeam` → server treats as `false` (default).
- `/compute-volumes` v0.1 shape (`peers` field with encrypted blobs) → legacy path still runs as today.

Result: v0.2.x clients keep working unchanged against a v0.3 server. v0.3 clients require a v0.3 server (a v0.3 client talking to a v0.2 server would just see "toggle does nothing"). **Deploy server first; release client after.**

### Client orchestrator

`positionTickInner` simplifies — client no longer team-filters locally:

```
SCANNING: applyPeerVolumes({ ally1: 1.0, ally2: 1.0, ... })   // unchanged
hold > 2s OR position invalid: return                          // unchanged
sendCoords(x, y, hearCrossTeam)
result = computeVolumes(pos, roomId, name)
applyPeerVolumes(result.peerVolumes)                           // server already filtered
```

### Settings UI

New third row in the existing Settings panel, after "Hide IP (Force TURN)":

```
Hear enemies at full vision range  [○ off]
  Default: only enemies within auto-attack range.
  Toggle on for full-vision-range hearing.
```

State persisted in `localStorage` under `lolproxchat.hearCrossTeam` (default `false`). Read on session start, sent in the `join` message and on every subsequent `coords`.

### Threat-model implications

- Default config no longer leaks enemy presence beyond stock-game visual range. The "binary: enemy is within 1200u" leak from v0.2.x becomes "binary: enemy is within 600u," which is effectively zero new information (a 600u-away enemy is already within auto-attack range and visually obvious in most line-of-sight scenarios).
- Modified clients can no longer bypass the filter — the server simply does not include out-of-toggle volumes in the response.
- Threat-model mitigations table gains two rows applied, the "what the design protects" section gains a paragraph about the default-config bound.

## 2. Issue #11 — Debug thumbnail dynamic panel resize

**Problem.** Overlay window is fixed at `height: 400, resizable: false` in `tauri.conf.json`. Debug thumbnail at `width: 100%` of the 240px panel ≈ 240px tall, overflows window. When Debug is toggled off the window doesn't shrink back.

**Fix.** Rust `resize_overlay(height)` command + JS that measures `panel.scrollHeight` and calls it on relevant events (Debug toggle, Settings toggle, ResizeObserver on thumbnail, peer-list updates).

Critical detail: the Rust command must also update `PANEL_HIT_RECT` so clicks below the panel pass through to the game after a shrink — `set_panel_size` already does this for width changes when the panel collapses, the new command needs the height equivalent.

Capability grant required in `src-tauri/capabilities/default.json` (verify the v2 permission name before implementing — Tauri 2 silently denies window resizing without it).

Extract `computeDesiredHeight(scrollHeight)` as a pure function so the clamp + breathing-room math is unit-testable without spinning up a real DOM.

## 3. Issue #1 — PTT global hook + rebindable PTT

**Problem.** `tauri-plugin-global-shortcut` uses Win32 `RegisterHotKey`, which LoL's DirectInput / raw-input layer intercepts before the global-hotkey API sees the press. F8 registers but never fires in-game.

**Fix.** Replace with `SetWindowsHookExW(WH_KEYBOARD_LL)`. Same technique Discord/Mumble/OBS use.

### Architecture

New `src-tauri/src/global_keys.rs`:

- Two atomic `u32`s holding the VK codes for PTT and toggle-mute.
- One hook proc registered with `WH_KEYBOARD_LL`.
- Hook proc: on key match, post to a `tokio::mpsc` channel and return immediately. **All work inside the hook is O(1) and lock-free.** A worker task drains the channel and emits Tauri events.
- `set_ptt_key(vk)` and `set_toggle_key(vk)` Tauri commands.

### Hook lifecycle gotchas

- `SetWindowsHookExW` must be called from a thread with a Win32 message pump (`GetMessage` loop). Tokio worker threads do not qualify. **Spawn a dedicated `std::thread` with a manual message pump for hook ownership.**
- Hook proc deadline is ~300ms by default (configurable via `LowLevelHooksTimeout` registry key). Slow callbacks get the hook silently uninstalled. Every code path in the proc must be fast.
- Antivirus / SmartScreen may flag low-level hooks as keylogger behavior. Risk mitigation: clear self-attestation in README + release notes ("low-level keyboard hook for in-game PTT, same technique as Discord/Mumble/OBS"). Monitor VirusTotal scores post-release.

### Caps Lock LED workaround

Default PTT key is **Caps Lock**. To prevent the LED toggling every press, the hook proc sends a synthetic `SendInput(VK_CAPITAL, down+up)` after acting on a real Caps Lock event, flipping the LED back. Standard Discord pattern; ~5 lines of code.

### JS — Settings UI

New "Input" section in Settings:

```
Push-to-Talk key:        [Caps Lock]      [Change…]
Toggle self-mute key:    [(unbound)]      [Change…]
```

`captureKey` flow: button click → "Press a key…" → next `keydown` captured → `browserKeyToWin32Vk(e.code)` → `localStorage.setItem(...)` → `sendToBackground('setPttKey', { vk })` → button text updates to `humanizeVk(vk)`.

`browserKeyToWin32Vk` and `humanizeVk` are pure mapping functions — unit-test in isolation.

Reject obviously hostile bindings: Esc, Tab, Q/W/E/R (LoL abilities). Show a warning, don't accept.

### Lifecycle

On app startup: read `localStorage`, send `set_ptt_key` + `set_toggle_key` to Rust. If localStorage empty, defaults are Caps Lock for PTT and unbound for toggle-mute.

Remove `tauri-plugin-global-shortcut` from `Cargo.toml` and `tauri.conf.json`. Existing JS event listener on the `'ptt'` event keeps working (Rust emits the same event name from the worker task).

## 4. CV tracking code tweaks

Driven by specific failure modes in IXAM's v0.1.33 logs (issue #7):

### Tweak 1 — cap continuous-hold duration at 5s

Today: holds can extend to 44s (`Resumed tracking after hold (44.37s)`), during which `extrapolatePosition` keeps extending the search box. By 44s the player is plausibly anywhere on the minimap — extrapolation is worse than re-acquisition.

Change: after 5s of continuous hold, force a drop back to SCANNING-style classifier-driven full-minimap scan. Add to `tracking.ts` state machine.

### Tweak 2 — classifier EMA recovery

Today: a couple of low classifier scores poisons the EMA for the rest of the session (IXAM's log: classifier ema stays at 0.00 for the whole game).

Change: use `max(currentEma * decay, latestRaw)` when the latest raw score exceeds the EMA, allowing a single confident hit to pull EMA back up. Change in `tracking-helpers.ts`.

### Tweak 3 — require classifier confirmation for LOCKED

Today: `SCANNING -> LOCKED via composite(score=0.42)` accepted even when classifier said "not your champion" (score 0.0). Result: LOCK on the wrong icon, immediate hold for 8+ seconds.

Change: require `classifierEma > 0.3` OR confident raw score on the candidate blob before accepting the SCANNING→LOCKED transition. If only composite agrees, stay in SCANNING and keep scanning. Add a gate in the LOCKED-transition decision in `tracking-helpers.ts`.

### Tweak 4 — don't broadcast unreliable positions

Today: `getHoldDurationSec() > 2` already early-returns from sending coords. But a LOCKED-via-composite-only state with `classifier ema === 0` is also unreliable and we shouldn't be sending those coords either.

Change: extend the early-return condition in `orchestrator.ts::positionTickInner` to also skip sending when `tracking.getClassifierEma() < 0.1` for the first few seconds after a LOCKED transition. Better to be inaudible than to broadcast fountain coords for 30s.

### Tests

All four tweaks are pure-function-ish changes inside `tracking-helpers.ts` (or testable extracts from `tracking.ts`). Add ~5-10 tests pinning behavior to specific scenarios reconstructed from IXAM's logs:
- 44s hold should trigger forced re-acquisition at 5s
- Classifier EMA poisoned to 0 should recover within 2-3 confident hits
- LOCKED-via-composite with classifier=0 should be rejected
- Coords-send should be gated by classifier confidence for first N seconds after LOCKED

Each test mirrors a log line we observed.

## 5. CV model retrain

### Phase 0 — feasibility check (BEFORE locking the v0.3.0 schedule)

Run the scrape + a 10-class smoke training to confirm:
- `scripts/update-champion-icons.js` still works against the LoL wiki structure
- `scripts/synthetic-data/` generators still produce sensible data
- `scripts/train_champion_classifier.py` is reproducible
- Wallclock for a small training run, extrapolate to estimate full 172-class time

If any of this is broken, model retrain slips to v0.3.1 and is queued for its own focused effort. CV code tweaks (Section 4) ship regardless — they are independent.

### Phase 6 — full retrain (parallel with the rest of v0.3.0 development)

1. `scripts/update-champion-icons.js` — full scrape, catches any new champs Riot added since the model was last trained.
2. `scripts/synthetic-data/` generators — produce fresh training data with realistic minimap-crop variation (blur, scale, lighting).
3. `scripts/train_champion_classifier.py` — full 172+ class training run. Long-running step.
4. Validation: spot-check 5-10 champions including the previously-mismatched Nunu & Willump / Dr. Mundo. Confirm >0.7 confidence on clean crops.
5. Lockstep check: if labels file changed (e.g., `"Nunu"` → `"NunuWillump"`), the `DISPLAY_TO_LABEL_NAME` map in `champion-classifier.ts` must update. The "every normalization target exists in label map" test from v0.2.1 will fail loudly if missed.
6. Ship: replace `models/champion_classifier.onnx` and `models/champion_labels.json` in the repo.

### Fallback

Keep the v0.2.1 model around as a backup artifact in git history. If post-ship feedback indicates the new model regresses real-game tracking, revert to the v0.2.1 model in v0.3.1 while we investigate.

## Release strategy

### Versioning

**v0.3.0** — minor bump under 0.x semver. Wire-protocol additions (`team` on join, `hearCrossTeam` on coords) are the canonical justification for the minor.

### Deploy order

```
1. Build + deploy v0.3 server (back-compat with v0.2.x clients — zero impact)
2. Verify server is serving v0.3 clients correctly (smoke test)
3. Build + tag + release v0.3.0 client (.exe asset, GitHub release)
4. v0.2.x users pick up via in-app updater
```

If retrain (Phase 6) finishes in time, the new model ships in v0.3.0. If not, v0.3.0 ships without it and v0.3.1 follows when ready.

### Estimated effort

- Tiered proximity (server + client): 1-2 days
- #11 thumbnail resize: 0.5 day
- #1 PTT hook + rebindable: 1-2 days
- CV code tweaks (Section 4): 1-2 days
- Model retrain (Section 5 wallclock): 1-2 days, mostly unattended

Total: ~1-1.5 weeks of focused work.

### Phase independence

Each of Sections 1, 2, 3, 4 is independently testable, mergeable, and reversible. Any one can slip to v0.3.1 without blocking the rest. Section 5 is gated by Phase 0 feasibility.

## Open questions / unknowns

- **Caps Lock PTT default** is good for not-conflicting-with-LoL-bindings, but some users find Caps-as-PTT physically annoying. Worth a brief mention in the release notes that they can rebind in Settings.
- **AV/SmartScreen impact of the low-level hook** is unmeasurable without shipping. Plan: monitor VT scan results post-v0.3.0; if scores degrade, add the README/release-note attestation language.

## What this design intentionally does NOT do

- Add per-team voice channels (allies-only voice room) — current "applyPeerVolumes returns ally=1.0" via the same WebRTC mesh already achieves the same thing without separate rooms.
- Add server-side rate limits specific to the new toggle — the existing `/compute-volumes` rate limit covers it.
- Build a CV eval harness — deferred to v0.4.0 (see "out of scope").
- Address any of #6, #7, #13, #15 directly — implicit improvements only.
- Touch the encryption back-compat path — the v0.1.x encryption path stays wired for as long as v0.1.x clients are connecting.
